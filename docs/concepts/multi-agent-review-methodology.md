# 多代理人審查系統 — 方法論反思

**入庫日期：** 2026-04-14
**類型：** 方法論 + 元認知
**相關案例：** `../concepts/japan-india-labor-analysis.md`

---

## 1. 問題起源

用戶要求對一份「在日印度勞工結構分析報告」進行數據驅動的審查與研究迭代。這份報告聲稱基於日本厚生勞動省和法務省統計，但多項核心數字無法驗證。

---

## 2. 框架設計

採用三代理迭代模型，理論基礎來自 Harness Pattern 的 Generator + Evaluator 分離原則：

```
┌─────────────────────────────────────────────┐
│           多代理審查系統流程                  │
├─────────────────────────────────────────────┤
│                                             │
│  ┌──────────┐    ┌──────────┐               │
│  │ Generator│───→│  Critic  │               │
│  │ 數據驗證  │    │ 偏誤審查 │               │
│  └──────────┘    └──────────┘               │
│        ↓              ↓                      │
│        └────────┬────────┘                   │
│                 ↓                            │
│          ┌──────────┐                        │
│          │Researcher│                       │
│          │政策評估   │                       │
│          └──────────┘                        │
│                 ↓                            │
│          結構化審查報告                        │
└─────────────────────────────────────────────┘
```

---

## 3. 執行過程中的問題

### 3.1 delegate_task 的 .context 傳遞混淆
嘗試使用 delegate_task 啟動平行子代理，但子代理的 context 被錯誤傳遞（吞入了其他任務的 context），導致任務失敗。

**教訓：** 當使用多個 parallel tasks 陣列時，每個 task 的 context 應完全隔離，不依賴 top-level context。

### 3.2 subagent exit_reason: max_iterations
子代理在 30 次迭代後耗盡，說明：
- 要麼任務過於複雜
- 要麼工具調用迴圈未正確終止

**最終採取了直接執行模式**（由主 agent 直接完成多代理分析），繞過了 subagent 的 context 傳遞問題。

---

## 4. 關鍵審查發現

### 4.1 最大的認知陷阱
本案例揭示了統計報告中最常見的偏誤：**數據來源虛構偏誤（Source Fabrication Bias）**。

分析者聲稱「基於官方統計」，但：
- 日本官方並未公佈「印度人的產業別分佈」
- 「印度人的簽證分佈」同樣不在官方統計中
- 這些精確數字是分析者自行估算的，卻未告知讀者

### 4.2 識別偏誤與使用未驗證數據的悖論
報告正確識別了 3 種偏誤（IT偏見、分類誤用、體感偏差），但這些識別本身是基於**未驗證的數據**做出的。

這形成了一個諷刺的邏輯鏈：
```
結論：印度人不只是IT（偏誤識別正確）
     ↑
支撐數據：產業分佈統計（數據未驗證）
     ↑
結論的可信度因此受損
```

---

## 5. 方法論原則（本次產出）

### 5.1 數據優先原則
> **當數據鴻溝大於聲稱精確度時，應擴大不確定性區間，而非填補估算值。**

本案例中，報告聲稱「IT佔30%」但數字根本不存在，正確的回應是：「置信度LOW，數據無法驗證」，而非姑息一個估算。

### 5.2 置信度傳染原則
> **低置信度數據支撐的結論，無論偏誤識別多麼正確，其結論置信度都會被傳染為 LOW。**

報告的偏誤識別工作本身是對的，但如果用未驗證數據來支持糾偏敘事，結論仍然不可靠。

### 5.3 數據存在性先驗原則
> **在聲稱數據之前，應先確認數據是否真正存在。**

這次案例最重要的發現是：印度人的國籍別產業統計在日本官方數據中並不存在。這意味著任何聲稱精確分佈的報告，在數據驗證面前都是LOW置信度。

---

## 6. 與 Harness Pattern 的關聯

Harness Pattern 的核心是「Generator 不能 judge 自己」，本系統延伸為：
- **Generator** = 數據驗證代理（提取聲明）
- **Evaluator** = 偏誤審查代理（識別問題）
- **Meta-Evaluator** = 政策評估代理（產出修正結論）

三者分離，避免單一代理既產出結論又為結論辯護。

本次執行後建立了 `research/multi-agent-labor-statistics-review` skill，可複用於：
- 外國人雇用統計報告審查
- 在台/在韓/在日外國勞工結構分析
- 移民/勞動力市場數據驅動審查

## 8. 後續優化方向

| 方向 | 說明 |
|-----|------|
| 強化 delegate_task context 隔離 | 為每個平行任務設計完全自足的 context |
| 增加「數據存在性預檢查」步驟 | 在多代理迭代前，先確認關鍵數據是否真的存在 |
| 置信度追蹤系統 | 在整個審查流程中追蹤每個聲明的置信度變化 |
